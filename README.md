# whitley

Private 2-node Kubernetes cluster using k3s, managed with Rancher, fully provisioned via Terraform.

## Architecture

```
192.168.x.10  (k3s-server)   — control-plane + Rancher UI
192.168.x.11  (k3s-agent)    — worker node
```

Both nodes are LAN-only. No public IP exposure. No port forwarding.

## Repository Structure

```
whitley/
├── .gitignore
├── scripts/
│   ├── install-k3s-server.sh     # uploaded + executed by Terraform on the server node
│   └── install-k3s-agent.sh      # uploaded + executed by Terraform on the agent node
└── terraform/
    ├── main.tf                   # root module — providers, module wiring
    ├── variables.tf
    ├── outputs.tf
    ├── terraform.tfvars.example  # copy to terraform.tfvars and fill in values
    └── modules/
        ├── k3s/                  # null_resource bootstrap via SSH remote-exec
        └── rancher/              # cert-manager + Rancher via Helm
```

## Prerequisites

The following tools must be installed on your workstation before running anything.

### Terraform

Version 1.6.0 or newer.

```bash
# macOS
brew install terraform

# Linux (via tfenv)
git clone https://github.com/tfutils/tfenv.git ~/.tfenv
echo 'export PATH="$HOME/.tfenv/bin:$PATH"' >> ~/.bashrc
tfenv install 1.6.0
tfenv use 1.6.0

# Verify
terraform -version
```

### kubectl

Required for `make ui` and for any manual cluster inspection.

```bash
# macOS
brew install kubectl

# Linux
curl -LO "https://dl.k8s.io/release/$(curl -sL https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
chmod +x kubectl && sudo mv kubectl /usr/local/bin/

# Verify
kubectl version --client
```

### Helm

Required for Terraform's helm provider to initialise.

```bash
# macOS
brew install helm

# Linux
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# Verify
helm version
```

### SSH key access to both nodes

Terraform connects to both nodes over SSH using key-based authentication. Ensure:

- You have a private key with access to the server node
- You have a private key with access to the agent node
- Both nodes accept root login via those keys
- Both nodes are reachable from your workstation on port 22

Test connectivity before applying:

```bash
ssh -i <server_key> root@<server_ip> "echo ok"
ssh -i <agent_key> root@<agent_ip> "echo ok"
```

### Node requirements

Both nodes must have:

- A supported 64-bit Linux distribution (Ubuntu 22.04+ or Debian 12+ recommended)
- Internet access during provisioning (to pull k3s and Helm charts)
- Ports open between each other: `6443` (k3s API), `8472/UDP` (flannel VXLAN), `10250` (kubelet)
- `curl` installed

## Quick Start

### 1. Configure variables

```bash
cd terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your node IPs, SSH key path, and k3s token
```

Generate a strong k3s token:

```bash
openssl rand -hex 32
```

### 2. Deploy

```bash
make init
make apply
```

Terraform will:
1. SSH into both nodes and install k3s (server then agent)
2. Fetch the kubeconfig from the server and rewrite it for remote access
3. Deploy cert-manager via Helm
4. Generate a private CA (stored in tfstate) and create a ClusterIssuer
5. Deploy Rancher via Helm, using the private CA for TLS

### 3. Access Rancher

Add the following entry to `/etc/hosts` on your workstation (one-time setup):

```
<server_ip>    rancher.local
```

Then run:

```bash
make ui
```

This will open `https://rancher.local` in your browser. You will get a certificate warning due to the private CA — click through it, or install the CA cert (`terraform output ca_cert_pem`) into your browser/OS trust store.

Get the initial admin password:

```bash
kubectl get secret --namespace cattle-system bootstrap-secret \
  -o go-template='{{ .data.bootstrapPassword | base64decode }}' \
  --kubeconfig terraform/kubeconfig.yaml
```

When prompted for a Server URL, use `https://rancher.local` so all cluster nodes can reach Rancher.

## Security Checklist

| Control | Status |
|---|---|
| k3s API (`6443`) bound to private LAN IP only | Enforced via `--bind-address` |
| TLS SANs contain only private IP | Enforced via `--tls-san` |
| Traefik ingress on port 443 (LAN only) | Handles `rancher.local` routing |
| ServiceLB binds Traefik to server LAN IP | No public IP exposure |
| Rancher accessible via hostname only | Requires `/etc/hosts` entry on client |
| Private CA generated by Terraform | CA key never leaves tfstate |
| kubeconfig gitignored | `.gitignore` |
| tfstate gitignored | `.gitignore` — store securely |
| k3s token gitignored | In `terraform.tfvars` — `.gitignore`'d |

## Upgrading

To upgrade chart versions, change `rancher_chart_version` or `cert_manager_chart_version`
in `terraform.tfvars` and re-run `terraform apply`.

## State Management

The default backend is local (`terraform.tfstate`). This file is gitignored.  
For multi-operator or backup scenarios, configure a remote backend (e.g. S3 with SSE, Terraform Cloud)
in `terraform/main.tf` before running `terraform init`.
